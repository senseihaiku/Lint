### ayghri/i-have-adhd
- **Verdict:** STEAL-PATTERNS (this is the origin of the owner's existing skill; steal the parts beyond the chat rules)
- **What it is:** A cross-platform (Claude Code / Codex / Cursor / Gemini) plugin that reshapes an agent's *output* for an ADHD reader — action-first, numbered, state-restating, no preamble. Ten rules, MIT.
- **Signal:** ~8.4k stars, last commit 2026-07-22 (a day before this eval — very active), multilingual READMEs, a real eval harness. Not hype; a small, sharp, well-maintained artifact.
- **Unique mechanism:** it is purely a **surface-layer output contract** with an opt-in `SessionStart` always-on hook (`touch ~/.claude/.i-have-adhd-always` → hook strips the SKILL's frontmatter and injects the ruleset into every session) plus a **blind A/B/C eval rig** (`rubric.md` + `cases.jsonl` + `run_evals.py`) that gates changes on weighted correctness/autonomy/actionability/safety scores. Nothing in the prior research set ships an eval harness for its own prompt.
- **Stealable for seshat:** (1) The always-on hook pattern — a POSIX-`sh`, fails-open-exit-0, `${CLAUDE_PLUGIN_ROOT}`-relative `SessionStart` command that injects a markdown file's body as a system nudge; this is exactly how seshat's HARNESS should inject retrieval context (recent notes, open questions) at session start without a heavyweight MCP. (2) The **eval-gates-the-prompt discipline**: seshat's retrieval skills should ship a `cases.jsonl` of real "find X later" queries + a blind rubric, so retrieval quality is measured, not asserted. (3) The rule "restate state every turn / lead with the next action" is a good spec for how a *retrieval surface* should present results — answer first, provenance after.
- **Architecture:**
  - Single `SKILL.md` with `disable-model-invocation: true` (explicit `/i-have-adhd` invocation only) + a persistence clause that survives topic changes.
  - `hooks/always-on.sh` gated on a user-created flag file; resolves `SKILL.md` relative to `$0`, strips YAML frontmatter with awk, prints the body as an injected preamble.
  - Platform adapters (`agents/gemini.toml`, `agents/openai.yaml`, `.cursor/skills/…`) generated/synced via a GitHub Action — one source, many surfaces.
  - `evals/` + `scripts/run_evals.py` + `tests/` — CI-checked prompt regression.
- **How it would plug into seshat:** Pure **HARNESS + SURFACE**. It touches no substrate and stores nothing — it is the model for seshat's thin surface: how retrieved knowledge is *presented*, and a hook template for auto-injecting retrieval context at `SessionStart`.
- **Answer to owner's note:** Confirmed — this repo *is* the origin of your output-shaping skill; the `/i-have-adhd` SKILL.md is verbatim what you're running. What it contains beyond a chat skill: (a) an opt-in always-on `SessionStart` hook so the shaping persists without re-invoking, (b) a real blind-eval harness that regression-tests the prompt, and (c) multi-platform adapters auto-synced from one source. Take the hook pattern and the eval rig; the rules themselves you already have.

### agenticnotetaking/arscontexta
- **Verdict:** STEAL-PATTERNS (borderline ADOPT for its retrieval layer — the single most retrieval-focused repo in this whole research effort)
- **What it is:** A Claude Code plugin that *derives* a complete personal-knowledge system (folders, schemas, skills, hooks, MOCs) from a 2–4 turn conversation, backed by a graph of **249 research claims** about tools-for-thought and agent cognition. Plain markdown + git substrate; no DB, no cloud.
- **Signal:** ~3.5k stars, v0.8.0, but last commit 2026-02-19 (~5 months stale as of this eval). Substantial real code (30+ skills, 4 hooks, 249 claim notes, kernel validation script) — the opposite of a marketing README. Momentum has cooled, but the artifact is deep and directly on-thesis.
- **Unique mechanism:** two things the prior set doesn't have. (1) **Derivation, not templating** — every structural choice traces to a cited research claim via `dimension-claim-map.md`, so the system can explain *why* it is shaped the way it is. (2) A first-class **retrieval harness (`/ask`)**: a 3-tier knowledge base (WHY=research claims / HOW=guidance / WHAT=domain examples) with an explicit **query-classification table** (WHY/HOW/WHAT/COMPARE/DIAGNOSE/CONFIGURE/EVOLVE → primary+secondary tier), a `claim-map.md` routing index read *before* searching, and a `deep_search → vector_search → keyword → grep` fallback chain. Plus `/next`, a state-driven "surface the one most valuable action now" skill.
- **Stealable for seshat:** This repo is a goldmine for the retrieval bottleneck:
  1. **The `/ask` query-classification router.** Before retrieving, classify the question type and route to the right index tier with a named secondary. Seshat should copy this table wholesale — it converts "search everything" into "search the right thing," which is the actual retrieval win.
  2. **`claim-map.md` as a routing index read *first*.** A cheap, hand/LLM-maintained "which topics live where" map that prevents blind semantic search. This is the same cost-saving idea as the llmwiki `_context.md` folder stubs, generalized into a retrieval router.
  3. **Descriptions as retrieval *filters*, not summaries** (claim: `descriptions are retrieval filters not summaries`). The YAML `description` field exists to let an agent decide *whether to load* a note — shorter, distinctive, retrieval-optimized — not to compress content. Directly actionable for seshat's frontmatter spec.
  4. **`retrieval utility should drive design over capture completeness`** — this is literally seshat's thesis, already written as a cited claim with corollaries (flat > hierarchical, topological > temporal, "how will *any future agent* read this" as an auditable exit-velocity metric). Steal the claim and its reasoning chain.
  5. **`/next` + `session-orient` hook = proactive retrieval.** Retrieval isn't only pull; the `SessionStart` hook injects workspace tree + goals + recent learned-methodology notes + maintenance signals, and `/next` computes 14 vault-state signals → one ranked action with a `next-log.md` dedup so ignored suggestions decay. Seshat's "get knowledge out at the moment it's needed" is partly a *push* problem, and this is a working template.
  6. **Confidence/status fields on claims** (`speculative→emerging→supported→established`, `status: archived`) and the honesty standard ("say when the graph doesn't cover it") — mirrors Hindsight's confidence-scored beliefs but in plain markdown.
- **Architecture:**
  - **Three-space vault:** `self/` (agent identity/methodology, slow-growth), `notes/` (the knowledge graph, the point), `ops/` (queue/sessions/observations/tensions). Names adapt per domain; separation is invariant.
  - **Fresh-context-per-phase pipeline (the "6 Rs"):** Record→Reduce→Reflect→Reweave→Verify→Rethink, each phase spawned as its own subagent (`/ralph` queue orchestration) so attention never degrades in a filled context window.
  - **Four hooks:** `session-orient` (SessionStart: inject state + capture session), `write-validate` (PostToolUse schema enforcement), `auto-commit` (async git), session-capture (Stop). Deterministic checks pushed out of the context window into shell.
  - **Retrieval:** ripgrep + MOC traversal by default; optional `qmd` MCP (Tobi Lütke's embedding tool) for semantic search — explicitly *optional*, matching seshat's "machine layer is derived, never source of truth."
  - **Knowledge base:** 249 claim notes as atomic markdown with propositional wikilinks (title-as-claim, so traversal = reasoning), `kernel.yaml` of ~12–15 non-negotiable primitives validated by `validate-kernel.sh`.
- **How it would plug into seshat:** Touches **all four layers**, and maps almost 1:1 onto seshat's stack. SUBSTRATE = the three-space markdown+git vault. MACHINE LAYER = optional `qmd` embeddings + the MOC/claim-map indexes (rebuildable, not source-of-truth). HARNESS = the 6-R skills, `/ask` router, `/next`, and the four hooks. SURFACE = `/ask` answer synthesis (answer-first, cite claims, flag tensions). The cleanest single steal is the `/ask` + `claim-map.md` retrieval router as seshat's HARNESS retrieval skill.
- **Answer to owner's note:** Worth the "must." This is the most retrieval-first artifact across the entire research effort — it even ships your thesis as a cited claim ("retrieval utility should drive design over capture completeness"). But note the trap that matches your 30 prior failures: its 6-phase pipeline, 4 hooks, schema enforcement, and 249-claim engine are a heavy *capture/structure* apparatus — exactly the over-investment that killed past attempts. Don't adopt the whole plugin. Extract the retrieval layer (`/ask` classification router, `claim-map` routing index, descriptions-as-filters, `/next` push-surfacing, confidence fields) and leave the derivation engine behind. Also weigh the 5-month-stale signal: treat it as a pattern library to mine, not a dependency to install.

### Batch synthesis
Both "musts" are correctly flagged, and together they cover seshat's two thin ends: arscontexta is the **retrieval brain**, i-have-adhd is the **surface skin** and the **eval discipline**. The most important finding is that arscontexta has already articulated seshat's core thesis as a research claim *and* built a concrete retrieval router around it (`/ask`'s classify-then-route-to-tier + a `claim-map.md` read before any search) — this is a better answer to "getting knowledge back out" than anything in the prior set (Basic Memory, Graphlit, even Hindsight), because it's plain-markdown and rebuildable rather than a database. The cross-cutting warning is that arscontexta itself over-invests in capture/structure (6-R pipeline, 4 hooks, 249-claim derivation engine) — the exact failure mode the owner is trying to escape — so it should be *mined, not installed*. **arscontexta deserves the deepest follow-up:** specifically a focused read of `reference/kernel.yaml`, `reference/semantic-vs-keyword.md`, and the `claim-map.md` routing index, to lift its retrieval layer into seshat's HARNESS while discarding the derivation machinery.
