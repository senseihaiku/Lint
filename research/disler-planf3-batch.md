### disler/planf3
- **Verdict:** STEAL-PATTERNS (with a caution flag — see owner's-note answer)
- **What it is:** A single "meta-skill" (one Claude Code skill that produces other artifacts) that templates every engineering plan into a self-contained HTML document with a synced visual identity, embedded generated diagrams, phased checklists, and validation loops.
- **Signal:** 121 stars, 33 forks, created 2026-06-21, single push (a polished launch drop tied to a YouTube breakdown), Fable-5-era. Real code, tight and self-contained, not hype — but young and thin (a 240-line SKILL.md + five ~10-line workflow files + two image scripts). It is a *pattern demo*, not a maintained system.
- **Unique mechanism:** The **append-only living-document metadata header with bidirectional back/forward references**. Every plan carries `created / modified[] / commits[] / agent[] / session[] / back_refs / forward_refs`, all comma-lists that may only ever be appended to. In the real sample, `back refs` points at both a local prompt file *and* a GitHub repo with specific file paths (`github:disler/pi-vs-claude-code (extensions/coms-net.ts, ...)`). Plans become nodes in a hand-maintained citation graph, and a dedicated `update-references.md` workflow keeps links reciprocal. The prior research set has append-vs-revise memory (Hindsight) and wikilink graphs (nashsu/llm_wiki), but not this "every artifact is a self-provenancing node whose edges are maintained by a first-class workflow" pattern.
- **Stealable for seshat:** (1) The **append-only provenance header** stamped on every derived artifact (which agent, which session, which commit produced/touched it) — this is a retrieval accelerator, because it lets you answer "where did this claim come from / what's the freshest version" without a separate index. (2) The **reciprocal-reference workflow** (`update-references`) as a template for a cheap, deterministic cross-linker that keeps a retrieval graph consistent. (3) The **meta-skill discipline**: one skill enforces identical section structure forever, so an agent (or you) always knows *exactly where* a decision/note/amendment lives in any artifact — retrieval-by-convention instead of retrieval-by-search. (4) **HTML-first self-contained output** as a SURFACE that renders anywhere and stores nothing.
- **Answer to owner's note ("THIS is a MUST"):** It is genuinely the best-crafted artifact in the batch, but be clear-eyed about *why* it's a MUST for seshat. Planf3 is, at its core, a beautiful **capture-and-structure** engine — the exact category the owner has over-invested in and abandoned 30 times. Adopting it wholesale would repeat the failure. The MUST-steal is narrow and it is about *retrieval*: the provenance header + reciprocal-reference graph + structural consistency are what make artifacts findable later. Take those three mechanisms; leave the gpt-image-2 diagram generation, the HTML template maximalism, and the `Performance > Speed > Cost` "spend tokens" ethos, which serve planning aesthetics, not retrieval.

**Architecture (how planf3 actually works):**
- Entry point `/planf3 "<prompt>" [questionable]` → `.claude/skills/planf3/SKILL.md` reads the prompt and *routes* to one of four workflows by intent (create / update / update-references / build), plus one image subworkflow.
- `workflows/create-plan.md` (9 steps): analyze → explore codebase (`AI_DOCS/`, `APP_DOCS/`) → fill the HTML **Plan Template** embedded in SKILL.md, replacing every `{{PLACEHOLDER}}` and expanding `<!-- repeat -->` blocks → generate images in parallel → save to `specs/<kebab>.html` → open in browser.
- The Plan Template is one self-contained HTML file: a `:root` CSS palette (the "synced visual identity"), a `<details>` metadata header, Purpose/Problem/Solution, Relevant Files (tagged existing vs new), Implementation Phases (per-task `[]/[wip]/[x]/[f]` status markers + per-phase Testing Strategy + a "do not exit until every box passes" validation loop), a global Validation section, an optional Questionables Q&A, a free-form Notes canvas, and an append-only Amendments log.
- `workflows/build-plan.md`: a *fresh* agent reads the whole plan (every image, every back-ref at depth 1), executes phases top-to-bottom, flips status markers in the file as it goes, loops each phase's tests until green, then appends timestamp/agent/session/commit to the metadata header. The plan file is the shared state between the planning agent and the building agent.
- `workflows/update-plan.md` / `update-references.md`: surgical in-place edits that never overwrite metadata (append-only) and always record an Amendments entry; references are kept bidirectional across both files.
- `scripts/generate_gpt_image.py` + `edit_gpt_image.py`: `uv` single-file scripts calling `gpt-image-2` (needs `OPENAI_API_KEY`), wide format, <10 words per image, matched to the `:root` palette.
- Net: it is a *stateful markdown/HTML skill* where the artifact itself is the database, the queue, and the log — no external store.

**How it would plug into seshat (per layer):**
- **SUBSTRATE (md + git):** Adopt the append-only provenance header as YAML/frontmatter on seshat's markdown notes (`created`, `modified[]`, `agent[]`, `session[]`, `commits[]`, `back_refs`, `forward_refs`). Git already gives you history; this makes provenance queryable *without* leaving the file. Keep it markdown, not HTML — HTML is planf3's surface choice, not a substrate requirement.
- **MACHINE LAYER (derived indexes):** The `forward_refs`/`back_refs` edges are the seed for a rebuildable link graph — but per seshat's own rule, don't treat the hand-written refs as source-of-truth; *derive* the reverse index by walking `back_refs` across all notes (like `update-references.md` does, but batch and automated), so the retrieval graph is always reconstructable from substrate.
- **HARNESS (skills/hooks):** Steal the **meta-skill + workflow-routing** shape directly: one seshat skill whose single job is to stamp every note/artifact into a fixed retrievable structure, and a companion `update-references`-style skill run as a hook to keep the link graph reciprocal on every write. This is the highest-value transplant — it turns retrieval into "look in the known place" rather than "search and hope."
- **SURFACE (thin, never stores):** planf3's "self-contained HTML that opens in a browser and holds its own state" is a clean model for a seshat read-surface — render a note (or a query result) to a throwaway single-file HTML view with its provenance header and its back/forward links as navigable anchors, storing nothing.

### disler/install-and-maintain
- **Verdict:** INSPIRATION-ONLY
- **What it is:** A tiny reference repo demonstrating the "deterministic script + agentic oversight" pattern: a `justfile` and two Python hooks (`setup_init.py`, `setup_maintenance.py`) run the *same* setup script, optionally supervised by a `/install` or `/maintenance` agent prompt that can ask clarifying questions.
- **Signal:** 99 stars, 38 forks, created & pushed Jan 2026 (frozen since). Small, deliberately minimal teaching artifact, not a product. Real but scoped to onboarding/CI, not knowledge work.
- **Unique mechanism:** The **three-rung ladder over one source-of-truth script** — `cldi` (hook only, deterministic/CI), `cldii` (hook + agent reports status), `cldit` (hook + agent asks questions first). "The script is the source of truth; the difference is only whether an agent supervises and whether it interviews you." Prior research (IndyDevDan's ADW/software-factory) has the deterministic-agentic framing already; this is just its smallest concrete embodiment.
- **Stealable for seshat:** The **"one deterministic action, three supervision levels" ladder** applied to index rebuilds. Seshat's MACHINE LAYER is "derived, rebuildable, never source-of-truth" — the same rung pattern fits perfectly: `rebuild` (deterministic, CI/cron), `rebuild + agent audits the diff and reports drift`, `rebuild + agent interviews you when it hits an ambiguous merge`. It cleanly enforces the substrate-is-truth / machine-is-rebuildable boundary.
- **Answer to owner's note:** n/a (no note). Worth 20 minutes for the `justfile` + hook pattern only; there is nothing here about retrieval.

### disler/claude-code-hooks-multi-agent-observability
- **Verdict:** STEAL-PATTERNS
- **What it is:** A real-time observability system that captures Claude Code hook events (all ~12 lifecycle events) via `send_event.py` hooks → HTTP POST → Bun/SQLite server → WebSocket → Vue dashboard, so you can watch every tool call, subagent handoff, and session event across a swarm live.
- **Signal:** ~1.5k stars, 387 forks, created Jul 2025, last pushed Feb 2026 (the flagship of this batch, widely forked/used, actively maintained until recently). Real and battle-tested.
- **Unique mechanism:** **Turnkey hook-event telemetry** — a drop-in `.claude/` that emits a structured, timestamped, source-app-tagged event stream of *everything an agent does*, persisted to SQLite and streamable. The prior research set covers memory/retrieval substrates but none of them capture the *agent's own behavioral exhaust* as a first-class, queryable event log.
- **Stealable for seshat:** The **hook → structured event → SQLite** pipeline as seshat's *self-instrumentation for retrieval*. The owner's real pain is retrieval, and the highest-value retrieval signal is usage: which notes were actually read, which searches were run, which paths led to answers. Wire `send_event.py`-style hooks (PreToolUse/PostToolUse/SessionStart) to log every seshat query, every note opened, every search miss into a rebuildable SQLite event index (MACHINE LAYER). That gives you access-frequency ranking, "recently touched," and — critically — a **log of retrieval failures** (queries that returned nothing), which is the exact dataset needed to fix the retrieval bottleneck. Steal the hook scaffolding and the summarize-on-emit idea; skip the Vue dashboard unless you want a live SURFACE.
- **Answer to owner's note:** n/a (no note). This is the most directly retrieval-relevant repo in the batch despite looking like "just monitoring" — its value to seshat is the queryable behavioral log, not the pretty dashboard.

### Batch synthesis
The disler/IndyDevDan ecosystem — planf3, install-and-maintain, the multi-agent observability system, and the previously-researched fusion-harness/ADW — collectively says seshat's HARNESS should be **thin, deterministic-first, and self-instrumenting**, not a new store. The recurring pattern across all four is *one deterministic action wrapped in optional agent supervision, with the artifact/log itself as the only state* (planf3's HTML plan is its own database; install-and-maintain's script is its own source-of-truth; the observability hooks turn agent behavior into a queryable event stream). For a system where **retrieval, not capture, is the bottleneck**, the stealable core is: (1) a meta-skill that stamps every note into a known, findable structure so retrieval is convention not search (planf3); (2) an append-only provenance header + reciprocal reference graph derived into a rebuildable index (planf3 + seshat's own machine-layer rule); and (3) hook-based telemetry that logs every query, hit, and *miss* so retrieval quality becomes measurable and fixable (observability). Crucially, the batch also carries a warning: planf3 is a gorgeous capture/structure engine, exactly the tar pit that killed the owner's 30 prior attempts — so take its *retrieval* mechanisms (provenance, reference graph, structural consistency) and consciously leave its token-maximalist planning aesthetics behind.
